Konzept & Anleitung

Vom Frontend-Spiel zur Multi-School-Lernplattform

Wie wir die Lernwelt behutsam zu einer mandantenfähigen Plattform ausbauen — mit Schüler-Accounts, Klassen-Clustering, Season-Steuerung und niederschwelligem Lehrer-Admin. Ohne den Charme zu verlieren, ohne Build-Toolchain-Wahnsinn, ohne von Tag 1 an für Millionen User zu planen.

Projekt: MPS Tablet-Schulung Stand: 28. Mai 2026 Skalierung: 1–5 Schulen, ≤ 500 Schüler Hosting: EU (DSGVO)

01Executive Summary

Wir verwandeln eine Frontend-only-Anwendung in eine echte Plattform — in 5 Phasen, mit minimalem Risiko und unter 50 €/Monat in der Pilotphase.

Was wird gebaut
Multi-Tenant SaaS
Eine Plattform, viele Schulen — jede mit eigenen Spielen, Seasons, Schülern.
Warum
Skalierbarkeit
Cloud-Sync, Lehrer-Dashboard, kein Code-Fork pro Schule, echte Auswertung.
Wie
Vercel + Supabase
Vanilla-JS bleibt. Supabase liefert DB, Auth, API, Realtime — alles aus einem Dashboard.
Was kostet's
0 – 50 €/Mo
Pilot kostenlos. Erste Schwelle ab ~25 $/Monat. Größere Infra erst bei 50+ Schulen.
Was bleibt
Der Charme
Spielordner, Kreaturen, Shop, Stilistik — alles unverändert. Nur der Storage-Layer wird ausgetauscht.

02Zielbild & User-Flows

Drei Rollen, drei Wege durch die Plattform — alle niederschwellig.

Schüler/in
„Ich will spielen."
  1. Webseite öffnen
  2. Schule aus Dropdown wählen
    (nur Schulen mit offenem Fenster)
  3. Nickname + 4-stelligen PIN vergeben
  4. Spielen — Fortschritt wird automatisch in der Cloud gespeichert
  5. Beim nächsten Mal: Schule + Nickname + PIN → weiter
Lehrer/in
„Ich starte eine Schulung."
  1. Login mit Email + Passwort
  2. Klasse anlegen (z. B. „7b 2026")
  3. Registrierungs-Fenster öffnen (90 min)
  4. Schüler registrieren sich live im Unterricht
  5. Live-Dashboard zeigt Fortschritt pro Schüler
  6. Seasons schalten, PINs zurücksetzen
Super-Admin (du)
„Ich verwalte das Ganze."
  1. Schule für eine Partnerschule freischalten
  2. Default-Spiele & Seasons konfigurieren
  3. Schul-Admins benennen
  4. Globale Statistiken einsehen
  5. Notfälle handhaben (Daten-Export, Lösch-Anfragen)

03Empfohlener Tech-Stack

Jede Komponente erklärt — mit Begründung und Alternativen.

Frontend-Hosting: Vercel (Frankfurt) Beibehalten

Das nutzt ihr bereits. Static-Hosting mit Edge-CDN, automatischer Git-Deploy, Preview-Deployments pro Branch. Kostenlos für Hobby-Use, Custom-Domain möglich. Region Frankfurt erfüllt DSGVO. Kein Wechsel nötig.

Backend + DB + Auth: Supabase EU Neu

Supabase ist eine Open-Source-Alternative zu Firebase. Es liefert in einem einzigen verwalteten Paket alles, was wir brauchen:

  • Postgres-Datenbank — relational, perfekt für unsere Schulen/Klassen/Schüler-Struktur.
  • Auto-API — jede Tabelle bekommt automatisch eine REST- und GraphQL-API.
  • Auth — Login per Email/Passwort out-of-the-box; PIN-Modell ergänzen wir per Edge Function.
  • Edge Functions — kleine Server-Endpunkte in TypeScript für Spezial-Logik (z. B. Coin-Buchhaltung, Anti-Cheat).
  • Realtime — Postgres-Änderungen werden via WebSocket gepusht; perfekt für Live-Lehrer-Dashboard.
  • Row-Level Security (RLS) — Multi-Tenancy auf Datenbank-Ebene: Lehrer sieht nur eigene Schule, Schüler nur eigene Zeilen.
  • Storage — falls später Bilder hochgeladen werden sollen.
Warum nicht selber bauen?
Ein eigener Node-/Python-Backend wäre 2–3 Wochen Mehraufwand, plus Wartung, plus Backups, plus Auth-Implementierung. Supabase erledigt das mit einem Klick und einer EU-Region. Wenn die Plattform wirklich groß wird, kann man Supabase später selbst hosten — der Code bleibt kompatibel.

Frontend-Code: Erstmal Vanilla bleiben Pragmatisch

Wir refaktorieren nicht nach React/Vue. Stattdessen bekommt creatures.js einen Storage-Adapter als zusätzliche Schicht:

// storage-adapter.js — neuer Layer zwischen Spielen und Persistenz
const StorageAdapter = {
  async save(gameId, data) {
    if (currentUser) {
      await supabase.from('game_progress').upsert({
        student_id: currentUser.id,
        game_id:    gameId,
        ...data
      });
    }
    // Fallback / Offline-Sicherheit:
    localStorage.setItem('lernwelt_v3', JSON.stringify(allData));
  },
  async load(gameId) { /* analog */ }
};

Die Spiele rufen weiter saveGameData() auf — nichts ändert sich für sie. Erst wenn ein Spiel mehr Backend-Features braucht (z. B. Multiplayer), führen wir Vite ein.

Admin-Panel: Separate HTML-Seiten

Zwei neue Pfade, gleiche Stilistik wie das Hub:

  • /admin/index.html — Lehrer-Dashboard
  • /admin/super.html — Super-Admin (nur du)

Beide nutzen den offiziellen @supabase/supabase-js-Client (via CDN, kein Build). Tabellen + Formulare reichen — kein eigenes Framework.

04Reicht Vercel? Eine ehrliche Antwort.

Kurz: Ja. Für jetzt.
Vercel ist nur euer Static-Host (HTML, CSS, JS, Bilder). Die Last bei einer Schul-Plattform liegt fast vollständig bei der Datenbank — und die ist bei Supabase.

Grenzen im Überblick

MaßstabWas hält Vercel + Supabase aus?Setup
1–5 Schulen, ≤ 500 SchülerLocker. Beides Free-Tier.0 €/Monat
5–10 Schulen, ≤ 2.000 SchülerVercel Free, Supabase Pro nötig (Backups, no-pausing).~25 $/Monat
10–30 Schulen, ≤ 10.000 SchülerFunktioniert, Vercel Pro für Custom-Domain in Org, Supabase Pro.~50–80 $/Monat
30+ Schulen, >10.000 SchülerEigene Infra erwägen (Hetzner + Coolify, self-hosted Supabase).~50–200 €/Monat + DevOps

Wann nicht mehr Vercel?

  • Wenn Schulen self-hosting fordern (manche Kommunen verlangen das).
  • Wenn ihr Lehrer-Daten in Deutschland braucht, nicht nur EU. (Supabase EU ist in Frankfurt/Stockholm — formal DSGVO ok, aber manche Schulträger sind strenger.)
  • Wenn ihr monolithische Backend-Logik aufbaut, die nicht in Edge Functions passt.

Bis dahin gilt: Vercel + Supabase reicht und ist die richtige Wahl.

Trennung der Verantwortlichkeiten:

  • Frontend (Vercel): nur Rendering + Spiellogik. Keine Geheimnisse, keine Buchhaltung.
  • Postgres + RLS: Wer darf was sehen — auf Datenbank-Ebene, nicht im Frontend.
  • Edge Functions: Coin-Buchhaltung, Anti-Cheat (Score-Plausibilität), Account-Anlage.
  • Realtime: Lehrer-Dashboard bekommt automatische Updates, wenn Schüler spielen.

06Datenmodell

Postgres-Tabellen — bewusst flach gehalten. Multi-Tenancy über school_id auf jeder Zeile + RLS-Policy.

TabelleWichtige SpaltenZweck
schools id, name, slug, registration_window_until, school_settings (JSONB) Jede Schule = ein Mandant. Registrierungs-Fenster und Customizing hier.
teachers id, school_id, email, role (admin/teacher) Lehrer mit echtem Account. Auth über Supabase Standard-Flow.
students id, school_id, nickname, pin_hash, last_login_at Pseudonyme Schüler. Kein Email, kein Klarname.
classes id, school_id, name, teacher_id Optional in Phase 1. „7b 2026" etc.
class_memberships student_id, class_id Verknüpfung — n:m, falls Schüler in mehreren Klassen.
seasons id, school_id, name, order, is_open Pro Schule eigene Seasons. Lehrer schaltet sie frei.
game_access school_id, game_id, status, password_hash, season_id Ersetzt config.js. Pro Schule eigene Spiel-Freigaben.
game_progress student_id, game_id, points, rounds_played, creature, growth Pro Schüler & Spiel eine Zeile. Ersetzt das lernwelt_v3-Objekt.
inventories student_id, data (JSONB) Shop-State (komplex, viele Felder) → als JSONB, ersetzt lernwelt_shop_v1.
audit_log id, student_id, event_type, payload, created_at Wer hat wann was getan — für Lehrer-Auswertung & Forensik.

Row-Level-Security in 3 Regeln

-- 1. Schüler darf nur eigene Zeilen sehen/ändern
CREATE POLICY student_own ON game_progress
  USING (student_id = auth.uid());

-- 2. Lehrer sieht alle Schüler seiner Schule
CREATE POLICY teacher_school ON students
  USING (school_id IN (
    SELECT school_id FROM teachers WHERE id = auth.uid()
  ));

-- 3. Super-Admin sieht alles (Custom Claim)
CREATE POLICY super_admin ON schools
  USING ((auth.jwt() ->> 'role') = 'super');

07Auth-Flow: Zeitfenster-Self-Signup

Genau wie besprochen — keine Zettelwirtschaft, niederschwellig, DSGVO-freundlich.

Lehrer-Seite

  1. Lehrer loggt sich ein (Email + Passwort, Supabase Auth Standard).
  2. Klickt im Dashboard „Registrierung öffnen" mit Dauer (z. B. 90 min).
  3. Supabase setzt registration_window_until = now() + 90 min auf der Schule.
  4. Dashboard zeigt Live-Countdown und neu hinzugekommene Schüler.

Schüler-Seite

  1. Schüler öffnet die Plattform.
  2. Sieht zwei Buttons: „Anmelden" und „Neu hier — Schule wählen".
  3. Beim Neu-Klick erscheint ein Dropdown — nur Schulen mit aktivem Fenster werden gelistet.
  4. Wählt Schule, gibt Nickname (frei wählbar) + 4-stelligen PIN ein.
  5. Edge Function prüft: Fenster offen? Nickname in dieser Schule frei? → Anlegen, JWT ausstellen.
  6. Schüler ist eingeloggt und kann spielen.

Wieder-Anmeldung

  1. Schule wählen → Nickname → PIN.
  2. PIN-Vergleich (gehashed, server-seitig). JWT zurück. Fertig.
PIN vergessen?
Lehrer im Dashboard: „PIN zurücksetzen" → Schüler darf einen neuen vergeben (analog Erst-Setup, aber ohne Fenster-Check, da Lehrer initiiert).
Sicherheits-Hinweis
Ein 4-stelliger PIN ist nicht bankensicher. Das ist Absicht: für Kinder-UX. Sicherheit kommt durch Rate-Limiting (max. 5 Versuche pro 15 min, dann Block) und durch Pseudonymisierung (kein Klarname → bei kompromittiertem PIN gibt's wenig zu holen).

10Was bleibt unverändert

Wichtige Beruhigung: Wir reißen nichts ein, was funktioniert.

  • Alle Spielordner unter GameHub/*/ bleiben unverändert
  • Die Funktionen saveGameData(), getGameData() in creatures.js behalten ihre Signatur — nur die Implementierung dahinter wechselt
  • Bestehende Nutzer mit localStorage können ihren Fortschritt beim ersten Login importieren
  • CSS-Stilistik, Kreaturen-Sprites, Shop-Mechanik — alles unverändert
  • Vercel-Deployment-Flow (Git push → automatisch live) bleibt gleich
  • Kein Build-Step in Phase 1 & 2 — du öffnest weiter HTML direkt im Browser zum Testen

11DSGVO-Checkliste

Pflichtprogramm für eine Schul-Plattform. Vieles davon ist durch die Architektur-Wahl bereits gelöst.

  • EU-Hosting — Vercel Frankfurt + Supabase EU (Frankfurt/Stockholm)
  • Auftragsverarbeitungs-Vertrag (AVV) mit beiden — Vorlagen liefern sie
  • Pseudonyme Accounts — Schüler haben Nickname, keinen Klarnamen
  • Keine Email-Adressen bei Schülern (nur bei Lehrern)
  • Datenschutzerklärung auf der Plattform (Stub liefere ich, anwaltliche Prüfung empfohlen)
  • Recht auf Löschung — Schule kann ihre kompletten Daten exportieren und löschen lassen
  • Transparenz — Schüler & Eltern sehen, was gespeichert wird (kurzer FAQ)
  • TOM-Doku (Technisch-Organisatorische Maßnahmen) — kurzes PDF, was wir tun
  • Backup-Verschlüsselung — Supabase macht das standardmäßig
  • HTTPS überall — Vercel macht das automatisch
Was wir nicht speichern
Keine IP-Adressen von Schülern in Klartext, keine Tracker, keine Cookies außer dem Auth-JWT, kein Google Analytics. Das macht die Datenschutzerklärung kurz und schmerzlos.

13Offene Fragen — vor Phase 1 klären

Diese Entscheidungen ändern Details, nicht die Architektur. Aber sie sollten vor dem Tabellen-Anlegen geklärt sein.

  • Klassen-Cluster: Brauchen wir classes schon in Phase 1, oder reicht school_id + ein Feld cohort_name auf students?
  • PIN-Länge: 4-stellig (kindgerecht) oder 6-stellig (sicherer)? Pro Schule einstellbar?
  • Lehrer-Sicht: Bekommt der Lehrer Einzel-Daten („Max hat 3 Spiele gespielt") oder nur Aggregate („17 von 22 haben Spiel 3 abgeschlossen")? DSGVO-Frage.
  • Easter-Eggs: Sollen Spiele wie das Atari-1337 pro Schule deaktivierbar sein?
  • Sprache: Englische UI für andere Bundesländer/Länder? (Vermutlich erst Phase 5+.)
  • Offline-Modus: Was passiert wenn die Schule kein WLAN hat? Cache-First-Strategie? Brauchen wir das?
  • Schüler-Wechsel: Wenn ein Schüler an eine andere Schule wechselt — Account migrieren oder neu anlegen?
  • Datenaufbewahrung: Wie lange behalten wir Daten nach einer Schulung? Schuljahres-Ende-Reset?
Nächster Schritt
Wenn das Konzept passt, klären wir diese 8 Punkte in einem kurzen Gespräch — danach kann Phase 0 (Supabase-Projekt + AVV) los.
